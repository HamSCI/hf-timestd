# hf-timestd Web UI - Monitoring Dashboards

**Real-time monitoring and visualization interface for hf-timestd.**

## ✨ Features

- **Real-time Monitoring** - System status, channel health, data quality
- **🔊 Live Audio Streaming** - Listen to any WWV/CHU channel directly from the browser
- **Carrier Analysis** - Carrier power and quality visualization
- **WWV/WWVH Discrimination** - 6-method analysis with differential delays
- **Timing Quality** - GPS/NTP timing status and metrics
- **Pure Presentation Layer** - No business logic, just displays data

---

## 🚀 Quick Start

### Prerequisites

1. **hf-timestd** installed and running
2. **Node.js 18+** and npm (or pnpm)
3. **Data being generated** by core recorder

### Installation

```bash
cd web-ui
npm install  # or: pnpm install
```

### Starting the Web UI

```bash
# From web-ui directory
npm start

# Or directly with custom data root
node monitoring-server-v3.js /path/to/data
```

Default data root is `/tmp/timestd-test` (test mode).

### Accessing Dashboards

**Entry point**: http://localhost:3000/ (redirects to summary)

**Active Dashboards**:
- **summary.html** - System overview, channel health, data completeness
- **carrier.html** - Carrier analysis and quality
- **discrimination.html** - WWV/WWVH 6-method discrimination
- **timing-dashboard-enhanced.html** - Timing quality metrics
- **gaps.html** - Gap analysis

## 🏗️ Architecture

**Design Principle**: Web UI is a **presentation layer only**.

- ✅ **Knows WHERE** data is located (via `timestd-paths.js`)
- ✅ **Knows HOW** to display data effectively
- ❌ **Does NOT know WHAT** data represents scientifically
- ❌ **Does NOT know HOW** data is generated

**All business logic resides in the analytics service**, not in the presentation layer.

**See**: `WEB_UI_ARCHITECTURE.md` for complete architectural details.

## 🛠️ API Endpoints

The server provides REST endpoints:

```
GET /api/v1/summary                          - All dashboard data
GET /api/v1/system/status                    - System processes
GET /api/v1/channels/status                  - Per-channel status
GET /api/v1/carrier/quality?date=YYYYMMDD    - Carrier metrics
GET /api/v1/channels/:name/discrimination/:date - Discrimination data
```

**See**: `PATHS_API_QUICK_REFERENCE.md` for complete API reference.

## 📊 Data Display

The web UI displays data generated by the analytics service:

### Summary Dashboard
- System status (core recorder, analytics, radiod)
- Channel health and completeness
- Data storage and upload status
- **🔊 Live Audio** - Click any channel to stream audio via ka9q-radio's `pcmcat` → WebSocket

### Carrier Analysis
- Carrier power and quality metrics (from Phase 2)
- Quality metrics (JSON from analytics service)
- Date navigation, frequency selection

### Discrimination Screen
- WWV/WWVH 6-method discrimination
- Per-method CSVs: BCD, timing tones, ticks, 440/500/600 Hz, test signals
- 7-panel visualization with method labels
- Ground truth from 14 minutes/hour with exclusive tones

### Timing Dashboard
- GPS/NTP timing quality
- time_snap events
- Timing annotations

## 🚨 Troubleshooting

### Server Won't Start
```bash
# Check if port 3000 is in use
lsof -i :3000

# Kill existing instance
pkill -f monitoring-server

# Check logs
tail -f monitoring-server.log
```

### Data Not Showing
```bash
# Verify paths are correct
node -e "import('./timestd-paths.js').then(m => console.log(new m.TimeStdPaths('/tmp/timestd-test').getDataRoot()))"

# Check if files exist
ls -lh /path/to/data/raw_buffer/

# Check API directly
curl http://localhost:3000/api/v1/summary | jq
```

### Path Sync Issues
```bash
# Validate Python/JavaScript path consistency
cd /path/to/hf-timestd
./scripts/validate-paths-sync.sh
```
