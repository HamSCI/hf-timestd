# SHARED_2500 -2906ms Anomaly Fix (2026-01-10)

## Issue Summary

SHARED_2500 (2.5 MHz) channel consistently produced D_clock = -2906ms to -3589ms, which was rejected as implausible (outside ±1000ms bounds), causing fallback to D_clock = 0.00ms with mode=UNK. This resulted in 0% detection rate for timing measurements on all 2.5 MHz broadcasts (WWV, WWVH, BPM).

## Root Cause

**Stale timing calibration with sample rate mismatch:**

The timing calibration file at `/dev/shm/timestd/state/timing_calibration.json` contained:
- `sample_rate: 20000` Hz (from old system configuration)
- `rtp_offset_samples: 586210` (learned when system was running at 20 kHz)

Current system operates at:
- `sample_rate: 24000` Hz (all channels)

### Impact of Mismatch

When calculating `expected_second_rtp` for D_clock computation:
```python
current_offset = rtp_timestamp % samples_per_minute  # 1,440,000 at 24 kHz
offset_diff = calibrated_offset - current_offset     # 586210 - 500640 = 85570
expected_second_rtp = rtp_timestamp + offset_diff    # WRONG by ~3587ms
```

The stale `rtp_offset_samples = 586210` caused `expected_second_rtp` to be calculated 86107 samples (3587ms) too large, resulting in:
```
D_clock = emission_rtp - expected_second_rtp = -3587ms
```

## Why Only SHARED_2500?

The stale calibration file only had an entry for SHARED_2500 with the incorrect sample_rate. Other channels either:
- Had no calibration entry (bootstrapping fresh at correct 24 kHz)
- Had correct 24 kHz calibration from recent operation

## Investigation Process

1. **Initial hypothesis**: RTP offset learning issue → ruled out (RTP timestamps correct)
2. **Second hypothesis**: Sample rate mismatch → ruled out (all channels use 24 kHz)
3. **Third hypothesis**: Bootstrap calculation error → ruled out (would affect all channels)
4. **Root cause identified**: Stale calibration with wrong sample_rate specific to SHARED_2500

### Diagnostic Logging Added

Added detailed D_clock calculation logging to `transmission_time_solver.py`:
```python
logger.info(
    f"D_clock calculation for {station} @ {frequency_mhz:.1f} MHz: "
    f"arrival_rtp={arrival_rtp}, propagation_delay={best_candidate.total_delay_ms:.2f}ms, "
    f"propagation_samples={propagation_samples}, emission_rtp={emission_rtp}, "
    f"expected_second_rtp={expected_second_rtp}, "
    f"emission_offset_samples={emission_offset_samples}, "
    f"D_clock={emission_offset_ms:.2f}ms"
)
```

This logging revealed the incorrect `expected_second_rtp` value and led to discovery of the stale calibration.

## Fix Applied

1. Deleted stale calibration files:
   ```bash
   sudo rm /dev/shm/timestd/state/timing_calibration.json
   sudo rm /var/lib/timestd/state/timing_calibration.json
   ```

2. Restarted analytics service:
   ```bash
   sudo systemctl restart timestd-analytics
   ```

3. System automatically re-learned calibration at correct 24 kHz sample rate

## Operational Impact

- **Duration**: Issue present since system configuration change from 20 kHz to 24 kHz (estimated Dec 26, 2025 based on stale calibration timestamp)
- **Channels affected**: SHARED_2500 only (WWV 2.5 MHz, WWVH 2.5 MHz, BPM 2.5 MHz)
- **Multi-station validation**: Degraded due to loss of 2.5 MHz measurements
- **Resolution time**: ~30 minutes investigation + immediate fix

## Lessons Learned

1. **Sample rate changes require calibration reset**: When changing system sample rate, all timing calibration files must be invalidated
2. **Diagnostic logging is essential**: The detailed D_clock calculation logging was critical for identifying the root cause
3. **Calibration validation**: Consider adding sample_rate consistency checks when loading calibration files

## Recommended Improvements

1. Add sample_rate validation when loading timing calibration:
   ```python
   if calibration['sample_rate'] != self.sample_rate:
       logger.warning(f"Sample rate mismatch: calibration={calibration['sample_rate']}, "
                     f"actual={self.sample_rate} - invalidating calibration")
       return None
   ```

2. Add SSRC tracking to detect recorder restarts (already implemented in timing_calibrator.py)

3. Consider adding calibration age checks (warn if >7 days old)
